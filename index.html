怎么阅读lua源码

Recommended reading order

  • lmathlib.c, lstrlib.c: get familiar with the external C API. Don’t bother with the pattern matcher though. Just the easy functions.
  • lapi.c: Check how the API is implemented internally. Only skim this to get a feeling for the code. Cross-reference to lua.h and luaconf.h as needed.
  • lobject.h: tagged values and object representation. skim through this first. you’ll want to keep a window with this file open all the time.
  • lstate.h: state objects. ditto.
  • lopcodes.h: bytecode instruction format and opcode definitions. easy.
  • lvm.c: scroll down to luaV_execute, the main interpreter loop. see how all of the instructions are implemented. skip the details for now. reread later.
  • ldo.c: calls, stacks, exceptions, coroutines. tough read.
  • lstring.c: string interning. cute, huh?
  • ltable.c: hash tables and arrays. tricky code.
  • ltm.c: metamethod handling, reread all of lvm.c now.
    You may want to reread lapi.c now.
  • ldebug.c: surprise waiting for you. abstract interpretation is used to find object names for tracebacks. does bytecode verification, too.
  • lparser.c, lcode.c: recursive descent parser, targetting a register-based VM. start from chunk() and work your way through. read the expression parser and the code generator parts last.
  • lgc.c: incremental garbage collector. take your time.

Read all the other files as you see references to them. Don’t let your stack get too deep though.
If you’re done before X-Mas and understood all of it, you’re good. The information density of the code is rather high.

Reference:

  2. Online Lua 5.1 source code browser

编译选项的含义[host, build, target]

Configure with —host, —target and —build options

When people configure a project like ‘./configure’, man often meets these three confusing options, which are more related with cross-compilation

  • —host: In which system the generated program will run.
  • —build: In which system the program will be built.
  • —target: this option is only used to build a cross-compiling toolchain. When the tool chain generates executable program, in which target system the program will run.

An example of tslib (a mouse driver library)

the dynamically library is built on a x86 linux computer but will be used for a embedded arm linux system.

An example of gcc:

A gcc compiler is build on x86 linux computer. [—build]
The gcc will be executed on an embedded arm linux system. [—host]
But the gcc will generate the binary program, which run in a x86 linux system. [—target]

ios远程调试

安装debugserver

ios默认是无法通过lldb调试的，默认只能通过xcode连接调试，有时候很不方便，能不能通过lldb远程连接调试呢？答案是可以的。
下面以ios8.4为例，详细说明具体配置过程:

  1. 首先iPhone需要越狱，越狱成功后安装OpenSSH服务。
  2. 安装ios8.4 SDK中的debugserver到iPhone上。
  3. 编写entitlements.plist配置文件。
  1. 使用codesign签名。
  1. 拷贝已签名的debugserver到iphone上。

调试程序

  1. 启动debugserver调试目标程序, debugserver支持直接启动程序和附加到程序进程进行调试，可以直接输入debugserver查看参数帮助。
  1. 启动lldb连接到远程debugserver开启的端口，进行调试。
  1. 通过wifi连接调试很慢，也可以通过USB进行调试。
  2. 安装usbmuxd。可以用来通过usb端口建立iphone的代理连接。
  1. 把iphone线接到电脑上，然后启动代理连接。
  1. 现在我们可以直接连接到本地1234端口来进行调试。

好了，进行享受lldb远程调试吧！

色彩理论(wiki摘要)

色彩概述

20世纪初色彩理论原则围绕着”纯的”或者说”理想”色彩而建立, 是用来描述感觉体验而非物理世界中色彩的属性。这导致原有的色彩理论原则并不能很好的适应现代的构想。

主要有下面3个问题:

  1. 色光混合(加色法)和色料混合(减色法)之间行为的困惑。这个困惑在于材料对光的吸收和眼睛对光的感知遵循不同的规则。

  2. 在描述从一个表面(如油漆或颜料)反射出来的色彩产生强烈亮度对比度这个重要效果时, 色料混合和色光混合是截然相反的; 色彩如棕色(browns)或赭色(ochres)无法出现在色光混合中。因此，中间值的黄色油漆和周围的亮白色之间的强烈亮度对比使得黄色呈现出绿色或棕色，而彩虹和周围的天空之间的强烈亮度对比使得黄色在彩虹中出现微弱的黄色或白色。

  3. 精确的描述色彩效果的需要。当大多数色彩效果都能以三个相对属性来定义反差时，就能以如”黄色”和”蓝色”之间反差构思出更通用的色彩。

    • 亮度(lightness)[light vs. dark, or white vs. black]
    • 饱和度(saturation)[intense vs. dull)]
    • 色相(hue)(如red, orange, yellow, green, blue or purple)

这些问题一直到19世纪末也没有解决。
